# 数据权限：拦截链与边界（M1 最小骨架 · 防返工）

> 适用范围：`hashmatrix-security` 数据资产可见性 / 行列管控的**最小骨架**约定。
> 目的：在 M1 先把**拦截链顺序**与**RBAC↔数据权限边界**落档，避免早期把「资产可见性」硬塞进模块
> RBAC 而在 SC-6 落地时返工。完整 SC-6 裁决引擎（主体×对象×动作×约束、行/列计划、动态脱敏）后置。
>
> 设计参考：主仓 `docs/architecture/05-多租户与控制平面.md` §5（schema-per-tenant + 行级兜底）；
> `prototype/docs/07-数据安全/03-统一数据权限（自研）.md` §8（拦截链、授权对象=元类实例）。

## 1. 两类权限，互不混淆（边界）

| | 功能权限（Function RBAC） | 数据权限（Data Permission） |
|--|--|--|
| 管什么 | 「能不能调这个**接口/动作**」 | 「能看到哪些**库表 / 行 / 列**」 |
| 落点 | `api` 层方法级 `@PreAuthorize`（`starter-security`，网关 `X-Roles` → `ROLE_*`） | **数据访问层**（租户 schema 路由 + 拦截链 ②③④） |
| 例子 | `@PreAuthorize("hasRole('security-viewer')")` 才能调 `/api/security/probe` | `security-viewer` 在本租户内仅可见 `sec_*.classification` / `sec_*.label` |
| 反例（禁止） | 把「某角色可见哪些表」写进 `@PreAuthorize` 表达式 | 把「能否调接口」塞进数据层 SQL |

> **铁律**：功能权限只判「主体 + 动作」，绝不内联「对象（库表/行/列）可见性」；后者一律走数据权限拦截链。
> 二者混淆是后续 SC-6 返工的最大来源——本仓在 M1 即明确分离。

## 2. 拦截链顺序（①→②→③→④）

数据访问在**租户隔离边界内**自上而下经四级，前级是后级的前提，**顺序不可调换**：

```
①  行级兜底（最高优先 · 不可绕过）   —— starter-tenant，已实现
        │   SET LOCAL search_path TO "sec_<tenant>", public（单事务内，池连接复用不串读）
        │   无租户上下文 → 兜底到 public（fail-safe，绝不跨租户串读）
        ▼
②  SC-6 行/列裁决接入点              —— 本次新增「占位」，M1 默认放行
        │   主体(角色) × 对象(库表) × 动作 × 约束 → 行计划 / 列计划
        │   M1：PassThroughAssetAccessPolicyAdapter（allow，// TODO SC-6），不附加行/列约束
        ▼
③  列管控（列级可见 / 屏蔽）          —— 后置（SC-6 产出列计划后落地）
        ▼
④  动态脱敏（按密级/标签改写取值）    —— 后置（SC-6 产出脱敏策略后落地）
```

### 各级职责与当前状态

- **① 行级兜底**——`libs-java/starter-tenant` 提供租户上下文；`infra/JdbcFlowableConnectivityAdapter`
  已示范 `SET LOCAL search_path` 范式（见其 javadoc）。这是**多租户隔离底线**，与业务规则无关、最高优先、
  不可被 ②③④ 绕过。**已实现，本次不动。**
- **② SC-6 接入点**——本次新增。端口 `domain/port/AssetAccessPolicy`（出站端口，六边形），
  M1 默认出站适配 `infra/PassThroughAssetAccessPolicyAdapter` 为**干净 pass-through**（恒 `allow`，带 `// TODO SC-6`），
  仅占位、不改变放行语义；行/列计划等字段随 SC-6 落地再加，**M1 不引入半成品 Optional / 临时兼容层**。
- **③ 列管控 / ④ 动态脱敏**——明确**后置**，本次不落代码，仅在链中标注位置，供 SC-6 接续。

## 3. SC-6 接入点（代码位置）

| 件 | 角色 |
|--|--|
| `domain/AssetRef` | 受管数据资产引用（库 `sec_*` 模板 + 表）；运行期 `sec_*` 绑定为 `sec_<tenant>` |
| `domain/AccessRequest` | 裁决输入：主体角色 × 对象(`AssetRef`) × 动作 |
| `domain/AccessDecision` | 裁决结果：M1 仅 `allowed` + `reason`；行/列计划随 SC-6 扩展 |
| `domain/port/AssetAccessPolicy` | **SC-6 接入点**（出站端口）：`decide(AccessRequest) → AccessDecision` |
| `infra/PassThroughAssetAccessPolicyAdapter` | M1 默认出站适配：恒放行（`// TODO SC-6`）。SC-6 落地时**替换为真实裁决适配**（同一端口），链上下游不改 |

> **为什么是 pass-through 而非 Optional/开关**：占位实现必须是**语义完整的「放行」**，而非「未实现」的空值。
> 这样接入点从第一天就有确定语义（allow），SC-6 落地只是**替换实现**（同一端口），不需要在调用方写
> 「若策略缺省则……」的临时兼容分支——杜绝过渡债。

## 4. 角色 → 可见库表（库表级角色门 · 示例映射）

`app/RoleVisibilityCatalog` 提供脱敏示例映射（M1 占位，SC-6 后由持久化的角色-资产绑定替代）：

| 角色 | 可见库表（当前租户 `sec_<tenant>` 内） |
|--|--|
| `security-viewer` | `sec_*.classification`、`sec_*.label` |

- 多角色取**并集**；未知角色 → 空集（**默认不可见**，fail-closed）。
- 这是「库表级」粒度的可见性闸门（链 ② 的对象维度输入），**不等于**功能权限——它回答「这个角色在数据层能
  看到哪些表」，而非「能不能调接口」。

## 5. M1 边界（明确后置，避免镀金）

- **不做**：完整 SC-6 主体×对象×动作×约束裁决引擎、行计划/列计划、动态脱敏全量、角色-资产绑定持久化。
- **做**：落档拦截链 ①→②→③④ 顺序与边界；SC-6 占位接入点（默认放行、干净 pass-through）；脱敏示例
  「角色→可见库表」映射 + 单测。
- **不变量（不可返工）**：① 行级兜底始终最高优先、不可绕过；② 功能权限与数据权限分离；③ SC-6 接入点为
  稳定端口，落地=替换实现而非改链。
